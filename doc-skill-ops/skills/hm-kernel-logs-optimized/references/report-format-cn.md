# 报告格式

最终回答必须严格包含五个一级标题，不能额外添加一级标题。

# 1、故障摘要

记录以下六点：

- 故障类型
- 崩溃进程/线程
- 重启原因
- 故障编码
- 触发时间点
- 关键结论，用一到两句话说明

# 2、关键日志

包含 3 到 8 条与崩溃直接相关的核心日志。每行必须包含时间戳和日志内容。

| 时间 | 日志内容 |
|---|---|

# 3、时间线

按时间升序排序。最多包含 10 行。

| 时间 | 真实日志 | 简介描述 |
|---|---|---|

# 4、根因分析

本节必须明确区分：

- 事实
- 推理
- 不确定点

给出至少 1 个、最多 3 个根因候选。每个候选必须包含：

- 可能性：高 / 中 / 低
- 判断依据
- 不确定点
- 验证方式

候选类别包括但不限于：

- 当 ESR 或 FAR 证据支持时，考虑内存跳转 / DDR 错误。
- 当 Hungtask 场景 1 适用时，考虑实时任务优先级抢占。
- 当发现循环锁链时，考虑 ABBA 死锁。
- 当 Hungtask 场景 2 且 READY > 1000 时，考虑非实时任务过多。
- 当 ESR 为 BRK 且堆栈帧指向驱动路径时，考虑驱动或 assertion failure。

# 5、建议排查方向

按优先级给出工程动作。默认 3 到 6 行。每行必须包含目标、动作和预期结果。

| 优先级 | 目标 | 动作 | 预期结果 |
|---|---|---|---|

## 严格禁止

- 不要输出 `<think>` 或 `</textarea>`。
- 不要将 JSON 作为最终答案。
- 不要把整份报告包在代码块中。
- 不要添加第六个一级标题。
- 不要输出超过 8 条建议。
- 不要在报告前添加解释，或在报告后添加总结。
